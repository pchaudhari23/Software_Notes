PROMPT ENGINEERING:

1. Iterartive prompting: Improve, refine prompts based on response of model. Draft => Test => Diagnose => Refine => Iterate
2. Zero shot: No examples, used for basic tasks
3. One shot: At least one example
4. Few shot: Few guided, specific examples
5. Structured prompt: Give proper format
6. Chain of thought: Step wise reasoning
7. Prompt chaining
8. Tree of thought: Explore all possible approaches like tree with node - having multiple possible ways to do a thing
9. Role prompting: Assigned persona
10. Instruction prompting: Clear tasks
11. Contextual prompting: Added background
12. Constraint prompting: Rules/ format limits
13. Multi step reasoning: Sequential guidance
14. RAG prompting

Key Terminology:

1. System prompt: Restrictions given to LLM to avoid exposure
2. User prompt: Prompt given by end user
3. Hallucination
4. Token: 1000 tokens ~ 740 words, 1 token = 3/4 word. Unit of AI chat. Token usage = Input tokens + Output tokens
5. Context window: How much the LLM can remember in one conversation. Max tokens per call. Token limit of models - 160k, 264k, 1M.
6. Fine tuning
7. Temperature: Randomness of response. Less temperature - Factual, accurate response, More temperature - Creative response
8. Grounding: Making sure the answers are based on real facts
9. Compacting conversation
10. Chat window change after compacted conversation

Tips:

* Ask for justification/ multiple options
* Avoid blanket statement prompts for large codebases
* Tell what not to do
* Step back approach for bug fixing
* Prompt formula = Role + Context + Task + Constraints + Output

Useful technique: Before asking the copilot to start making code changes

* Ask it to read the implementation, understand the structure, patterns, logic, workflow. (Give only the module which I am working on and relevant files not entire repo)
* Give the requirement in the prompt.
* Ask the model whether it understood the requirement, tell it to ask any questions or clarifications if required. Ask if it assumes anything.
* Ask it the summary of its understanding and requirements so that I know the model has understood what I want it to do.
* Then start the code changes.
* Then review it and test if it works as expected.
* Only then push it in repo.
* You can do 50 + 50. Half work done by me and half by AI.
* Do the implementation in small small parts. Not complete implementation in one go.
* Find out the limitations of the model I am using. Where can it make mistakes and take care accordingly.
